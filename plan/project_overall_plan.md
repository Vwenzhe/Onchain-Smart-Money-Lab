# AI 驱动的链上聪明钱研究展示站整体方案

## 1. 项目定位

### 1.1 项目名称

AI 驱动的链上聪明钱研究展示站

### 1.2 项目目标

基于现有的链上 SQL 分析能力，将当前的 `web3-smart-money-analysis` 项目从一组面向研究的查询脚本，升级为一个可复用、可迁移、可展示、可扩展的链上研究系统。

该系统第一阶段聚焦在：

- ETH 单链
- 多个预设 token 的统一分析模板
- 聪明钱候选池分析
- 成本价与持仓估算
- PnL 快照与收益结构拆解
- AI 地址画像
- AI 异动归因
- 简洁前端展示

### 1.3 第一阶段产品形态

第一阶段更偏向于一个：

- 公开展示站
- 自用研究工具
- 招聘作品集项目

其重点不是实时交易执行，而是：

- 方法论清晰
- 架构清晰
- 分析结果可解释
- 指标和 AI 输出可复用

## 2. 当前项目基础

当前项目已经具备较清晰的分析骨架，核心逻辑包括：

- 候选地址池筛选
- 成本价和持仓估算
- 明细校验
- PnL 快照

现有 SQL 的职责大致如下：

- `D3_candidate_pool_fet_eth_rndr_sol.sql`
  - 生成候选地址池和日度聪明钱信号
- `D4_cost_basis_fet_eth.sql`
  - 计算候选地址成本价、仓位价值和浮盈
- `D4_cost_basis_validation_fet_eth_rndr_sol.sql`
  - 进行地址日度状态和口径验证
- `D5_pnl_snapshot_fet_eth.sql`
  - 做收益分桶和群体 PnL 画像

这说明项目已经完成了从原始转账数据到研究结论的第一层闭环，下一步不是推翻重来，而是做：

- 单链收敛
- 模板化重构
- 特征层沉淀
- AI 解释层接入
- 展示层产品化

## 3. 项目核心价值

该项目的核心价值不在于“做一个会总结的 AI 页面”，而在于构建一条完整研究链路：

- 链上数据提取
- 聪明钱识别
- 成本与仓位估算
- 盈亏结构分析
- 地址标签与画像
- 新闻与链下信息归因
- AI 解释输出
- 页面化展示

项目真正的差异化来自：

- 你自己的候选池定义
- 你自己的成本和持仓估算逻辑
- 你自己的风险结构和集中度分析思路
- 你自己的 AI 提示词和解释框架

## 4. 总体架构

项目建议采用 5 层结构。

### 4.1 配置层

作用：

- 管理 token 元信息
- 管理参数
- 让分析逻辑模板化

建议管理的配置字段：

- `token_symbol`
- `token_name`
- `chain_name`
- `contract_address`
- `lookback_days`
- `min_active_days`
- `min_net_flow_usd`
- `price_symbol`
- `is_enabled`

目标：

- 后续接入新 token 时，只改配置，不改主分析逻辑

### 4.2 分析层

作用：

- 用标准化 SQL 生成结构化分析结果

建议拆分为固定的 4 个分析模块：

- `01_token_candidate_pool.sql`
- `02_token_cost_basis.sql`
- `03_token_position_validation.sql`
- `04_token_pnl_snapshot.sql`

该层的要求：

- 命名统一
- 输出字段统一
- 参数入口统一
- 每个 SQL 职责单一

### 4.3 特征层

作用：

- 从分析层结果中抽取稳定、可复用的特征
- 为前端和 AI 提供统一输入

建议产出的核心表：

- `token_signal_daily`
- `address_profile_snapshot`
- `token_risk_snapshot`

建议沉淀的特征：

- 价格
- 候选聪明钱地址数
- 候选净流入 USD
- 候选平均净流入
- 聪明钱平均买入成本
- 当前价格与平均成本偏离
- 浮盈地址占比
- PnL 分布
- Top 地址集中度
- 候选池扩张速度

### 4.4 AI 解释层

作用：

- 将结构化数据转成自然语言结论
- 把链上信号和链下信息拼接成可读分析

第一阶段建议只做两个 AI 功能：

- 地址画像
- 异动归因

不要第一阶段就做：

- 币价预测
- 自动交易建议
- 过度主观的“名人地址断言”

### 4.5 展示层

作用：

- 用简洁网页把分析结果组织起来
- 作为研究展示和作品集交付

建议页面结构：

- 顶部 Token 选择区
- 基础指标区
- 图表区
- AI 分析区

## 5. 功能边界

### 5.1 第一阶段做什么

- ETH 单链
- 预设若干 token
- 手动刷新数据
- 统一 SQL 模板
- 输出价格、仓位、PnL、聪明钱核心指标
- 地址画像
- 异动归因
- 简洁前端

### 5.2 第一阶段不做什么

- 双链统一分析
- 实时更新
- 自动交易信号
- 账户系统
- 权限系统
- 复杂搜索系统
- 大而全的 token 覆盖

## 6. AI 功能定义

### 6.1 地址画像

输入：

- 活跃天数
- 首次买入时间
- 持仓规模
- 成本价
- 当前浮盈
- 持仓时长
- 日度买卖节奏
- 净流入规模

输出：

- 地址标签
- 行为风格总结
- 风险提示
- 重点观察建议

目标：

- 让用户快速理解“这类地址是什么风格的钱”

### 6.2 异动归因

输入：

- 价格变化
- 聪明钱地址数量变化
- 候选净流入变化
- 成本与仓位结构变化

外部输入：

- 新闻 API
- 官方公告
- 交易所公告
- 社媒信号摘要

输出：

- 近期异动概述
- 可能催化因素
- 证据强弱
- 风险和不确定性说明

要求：

- 不直接下绝对因果结论
- 采用“可能”“疑似”“当前证据支持”的表述

### 7.1 页面结构

一个主页面即可，后续再扩详情页。

建议模块如下：

- Token 选择器
- Token 基础信息卡
- 核心指标卡
- 图表区
- 聪明钱地址摘要表
- AI 地址画像摘要
- AI 异动归因摘要

### 7.2 建议展示的核心指标

- 当前价格
- 聪明钱候选地址数量
- 候选地址净流入
- 候选地址平均净流入
- 平均买入成本
- 当前价格相对平均成本的偏离
- 浮盈地址占比

### 7.3 建议展示的核心图表

- 价格走势
- 聪明钱地址数量走势
- 聪明钱净流入走势
- 平均成本与价格对比
- PnL 分布图

## 8. 推荐技术栈

### 8.1 第一阶段技术栈

- 数据分析：SQL
- 数据接入：Dune API 或手动导出
- 后端：Python + FastAPI
- AI：OpenAI API 或 Anthropic API
- 前端：Next.js
- 图表：ECharts / Recharts
- 配置管理：JSON / YAML

### 8.2 第一阶段技术原则

- 优先简单可控
- 优先手动刷新
- 优先可演示
- 优先结构清晰

## 9. 实施路线

### 9.1 阶段一：单链模板化

目标：

- 统一 SQL 命名、参数、输出字段
- 做出一个可复用的 ETH token 分析模板

### 9.2 阶段二：特征层稳定化

目标：

- 抽出稳定特征
- 固定核心指标
- 为 AI 和前端提供统一输入

### 9.3 阶段三：AI 地址画像

目标：

- 规则标签化
- LLM 生成自然语言画像

### 9.4 阶段四：AI 异动归因

目标：

- 接入新闻与公告源
- 对链上异动生成解释摘要

### 9.5 阶段五：前端展示

目标：

- 将数据、特征、AI 分析整合到一个公开展示站中

## 10. 风险与注意事项

### 10.1 风险一：范围过大

容易出现的问题：

- 还没完成模板化就开始写前端
- 还没稳定指标就急着接 AI
- 同时做双链、Bot、网页、归因、问答

建议：

- 第一阶段严格做减法

### 10.2 风险二：AI 输出过度主观

容易出现的问题：

- 直接断言“名人地址”
- 直接断言“杀猪盘”
- 直接断言“价格上涨原因”

建议：

- 用证据链表达，不下绝对判断

### 10.3 风险三：数据层不稳定

容易出现的问题：

- 字段口径不一致
- 同义字段命名混乱
- 前端和 AI 各自消费不同版本数据

建议：

- 先统一特征层，再开发上层能力

## 11. 交付成果

第一阶段完成后，理想交付成果包括：

- 一套统一命名的 SQL 模板
- 一份 token 配置清单
- 一套稳定的特征层输出
- 一个 AI 地址画像功能
- 一个 AI 异动归因功能
- 一个简洁可展示前端
- 一份清晰的 README 和项目说明

## 12. 成功标准

如果第一阶段完成后满足以下条件，就说明方向是正确的：

- 新增一个 ETH token 只需改配置而无需重写主逻辑
- 页面可以完整展示一个 token 的核心研究视图
- 地址画像能稳定输出有解释力的文本
- 异动归因能给出结构化且克制的说明
- 项目可以作为公开作品集展示
- 你自己后续扩展时不再需要重新梳理思路
